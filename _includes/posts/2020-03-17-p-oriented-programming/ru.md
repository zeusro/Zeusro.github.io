## Неправильные способы написания кода

1. Оснастить комплексными механизмами внутренней проверки работоспособности
1. Иметь 2 или более `code reviewers` перед слиянием веток
1. Создать полные тестовые среды перед официальными обновлениями
1. Покрытие тестами выше 80%
1. Использовать grayscale для каждого обновления развертывания
1. Не использовать root для управления серверами операций
1. Нанять инженеров с солидными основами компьютера
1. Обеспечить 99.99% доступности для базовых зависимостей инфраструктуры (Redis, MySQL)
1. Не использовать `sudo rm -rf /` для завершения обновлений сервера

Если вы хотите стать **P-ориентированным разработчиком**, **P-ориентированным менеджером**, **P-ориентированным XX**, все эти дурные привычки должны быть удалены.

## Правильные способы для открытого исходного сообщества

Ваши собственные проекты с открытым исходным кодом должны подчеркивать характер "**озорства**". Проект должен быть интересным, например:
1. [Использование динамического программирования, чтобы сделать подлеца украсть чужого партнера](https://www.zeusro.com/2020/03/14/dynamic-optimization/)

1. Не нравится, что веб-сайт `dnspod` слишком сломан, [разработка плагина Tampermonkey для модификации dnspod](https://www.zeusro.com/2019/07/05/mydnspod/), и, наконец, отправка электронного письма в HR Tencent, чтобы пожаловаться на их дизайн продукта

### Правильные способы для Pull Requests

`pull request` также должен подчеркивать характер "**озорства**"

Помните, `pull request` — это ваша лучшая возможность отомстить `reviewers` (эти люди обычно ваши начальники). Идите на все, чтобы копать ямы в коде и документации. Например:

1. Используйте `int32` для переменных, которые должны быть `int64`, позволяя проблеме постепенно появляться со временем
1. Творческие орфографические ошибки
1. Максимально используйте набор символов `UTF-8`, используйте неанглийские буквы, символы ASCII
1. Модуляризуйте как можно больше, util, interface, service implement, web front-end, web back-end все сделаны в `git` `submodules`, и между каждым `submodule` есть ромбовидные зависимости. Кто знает, когда проект верхнего уровня web может работать. Больше проектов означает больше KPI.
1. Настаивайте на использовании программирования `Java` и нарушайте каждое правило в [Руководстве по разработке Java Alibaba](https://github.com/alibaba/p3c)
1. Не пишите никаких деталей, связанных с продуктом, в документации продукта (это для защиты вашего продукта от разрушения!)
1. Чтобы препятствовать любой тенденции нанимать внешних подрядчиков по обслуживанию, вы можете разбросать атаки и клевету против других компаний-разработчиков программного обеспечения в коде, особенно любых, которые могут заменить вашу работу

Самозапутывающий код предназначен для Бога. Хорошо защитите наш код. Даже если они попадут в руки конкурентов, мы совсем не паникуем. Пока баги, которые вы пишете, достаточно странны, компания не посмеет легко уволить вас! Конкуренты также будут уважать вас!

**P-ориентированное программирование** фокусируется на идеях, а не на результатах. Научитесь говорить ерунду серьезно и писать код, полный багов.
