## データベースパラダイムはETCDの核心

貧乏人レモネードは最終的に配信される製品であり、ユーザーリクエストのデータフローは分散システムの構築とは逆である。

情報システムにとって、データベース（ETCD）は基盤インフラである。そしてデータベースパラダイムはETCDの核心である。

ETCDはこのレシートを構造化ストレージ（object）の方式で保存している。

浩浩妈が注文した「貧乏人レモネード」は、ユーザーの「期待」（spec）を表している。

蜜雪冰城（Kubernetes）は、次の行動を通じて、この「貧乏人レモネード」を作成し、配信する。

## 現実と理想の差はアプリケーション配信の目標

浩浩妈の理想（spec）の「貧乏人レモネード」と実際（status）に手にした「貧乏人レモネード」は2つの異なる概念であり、商業的な表現では「画像は参考のみ」である。
美団、餓了麼などの出前プラットフォームがクラウドサービスプロバイダー（cloud provider）と見なされるなら、蜜雪冰城本店は配信パイプラインの次の環（API server）である。

```draw
┌───────────────────────────────┐
│        Control Plane          │
│ ┌───────────────────────────┐ │
│ │ kube-apiserver            │ │
│ │ etcd (data store)         │ │
│ │ kube-scheduler            │ │
│ │ kube-controller-manager   │ │
│ │ cloud-controller-manager  │ │
│ └───────────────────────────┘ │
└───────────────────────────────┘
                 │
                 ▼
┌────────────────────────────────────────────┐
│               Worker Node                  │
│ ┌────────────┐  ┌────────────┐  ┌────────┐ │
│ │ kubelet    │  │ kube-proxy │  │ runtime│ │
│ └────────────┘  └────────────┘  └────────┘ │
│               └──► Pod(s) (containers)     │
└────────────────────────────────────────────┘
```

蜜雪冰城本店（API server/Control Plane Node）が実際にタスクを実行する店舗（Worker Node）にタスクを発行し、
店長（kubelet）が具体的な実施タスクを割り当て、これらの店舗の具体的な従業員（Container Runtime）が最終的な製品制作を担当する。

## 最後の1キロは出前配達員の義務

製品が完成した後、出前配達員が最終的な配信を行う必要がある。出前配達員は実際には「貧乏人レモネード」（ビジネス）の監督者でもある。

![img](/img/in-post/i-love-haohao/202312252211598563.jpg)

輸送中、製品の完全性（可観測性分析）を確保し、最終的な配信と顧客の要求（期待）の一致を実現する必要がある。
